1. Introduction

The food delivery industry has witnessed exponential growth in recent years, driven by the convenience of ordering meals from the comfort of one's home. EatEasy is a robust, full-stack multi-vendor food delivery application designed to address this demand by connecting hungry customers with local restaurants through a seamless digital interface.

The primary objective of EatEasy is to provide a scalable and user-friendly platform that supports three distinct user roles: Customers, who wish to discover and order food; Restaurant Owners, who need tools to manage their menus and process orders; and Administrators, who oversee the health and integrity of the entire platform.

1.1 Scope

The scope of this project encompasses the development of a responsive web application that includes real-time order processing, secure authentication, dynamic menu management, and comprehensive analytics. It serves as a complete solution for small to medium-scale food delivery operations.

1.2 Core Modules

  • Customer Module: Empowering users to discover restaurants, filter by cuisine, view detailed menus, manage a shopping cart, place secure orders, and track delivery status in real-time.
  • Restaurant Owner Module: Providing owners with a comprehensive dashboard to manage restaurant profiles, edit menus (items, prices, images), update operating hours, and process incoming orders with status updates.
  • Admin Module: Integrating high-level management tools to verify new restaurant registrations, manage user accounts, oversee platform analytics, and ensure quality control across the service.

2. System Architecture

Interaction Workflow:

  6. Data Access: The Service layer invokes the Repository interface to perform CRUD operations.
  7. Database: Hibernate executes the optimized SQL query against the MySQL database.
  8. Response: The result set is mapped back to Entity objects and returned up the chain as a JSON response to update the UI.

2.2 Architectural Pattern

The project follows a Layered Architecture pattern to ensure separation of concerns, scalability, and maintainability.

  • Presentation Layer (Frontend): Handles UI rendering and user interactions using Angular components.
  • API Layer (Controller): Defines RESTful endpoints and handles HTTP requests/responses.
  • Business Layer (Service): Contains the core application logic and transaction management.
  • Persistence Layer (Repository): Abstraction layer for direct database interactions using Hibernate.
  • Database Layer: Relational data storage using MySQL.

3. Frontend Components (Angular)

3.1 Technology & Structure

The frontend is built using Angular (Latest Version), utilizing its modular design system. The application is structured into Feature Modules (`Auth`, `Admin`, `Customer`, `Restaurant`) to enable lazy loading and better code organization.

3.2 Authentication Module (features/auth)

  • Login Component: Captures user credentials and sends them to the backend. Upon success, it stores the received JWT token in local storage for session management.
  • Register Component: A dynamic form that adapts based on the selected role (Customer or Restaurant Owner), collecting necessary initial data like name, email, and phone.

3.3 Admin Module (features/admin)

  • Dashboard Component: The central hub for administrators to view platform-wide statistics such as total users, active restaurants, and total orders.
  • Restaurant Management: A table view allowing admins to approve pending restaurant registrations or block/unblock existing ones.
  • User Management: A list view of all registered users (Customers and Owners) with options to manage their accounts.
  • Coupon Management: A CRUD interface for creating discount codes (e.g., "SAVE20") with expiry dates and usage limits.

3.4 Customer Module (features/customer)

  • Home Component: The landing page featuring a search bar, promotional banners, and a list of top-rated restaurants. Implements filtering logic based on cuisine or name.
  • Restaurant Detail Component: Displays the full menu categorized by sections (Starters, Main Course, etc.). Users can add items to their global cart state from here.
  • Cart & Checkout: The Cart component calculates totals, taxes, and manages quantities. The Checkout component collects delivery addresses and finalizes the order.

3.5 Restaurant Module (features/restaurant)

  • Dashboard: Visualization of daily metrics including total orders, revenue, and active order counts.
  • Menu Management: A CRUD interface allowing owners to upload food images, set prices, and mark items as "Out of Stock".
  • Order Management: A real-time list of incoming orders. Owners can update status from "Pending" to "Preparing", "Ready", and "Delivered".
  • Restaurant Profile: A form to update restaurant details like name, description, address, and opening hours.

4. Backend Components (Spring Boot)

4.1 Security Configuration

The SecurityConfig class is the backbone of the application's security. It implements a Stateless Authentication mechanism using JWT (JSON Web Tokens).

  • SecurityFilterChain: Configures HTTP security to disable CSRF (unnecessary for stateless APIs), enables CORS to allow frontend communication, and defines public vs. protected endpoints.
  • JwtAuthenticationFilter: An internal filter that intercepts every request, extracts the "Authorization" header, validates the JWT token, and sets the SecurityContext for the current thread.
  • PasswordEncoder: Uses BCryptPasswordEncoder to hash passwords before storing them in the database, ensuring that raw passwords are never compromised.

4.2 Controller Layer (API)

Controllers act as the entry point for the backend. For example, the RestaurantController.java:

  • @RestController: Marks the class as a web controller handling JSON.
  • Endpoints:
    • POST /api/restaurants: registers a new restaurant.
    • GET /api/restaurants: accepts query parameters (search, cuisine) to filter results.
    • PUT /api/restaurant/status/{id}: allows owners to toggle their restaurant's open/closed status.

4.3 Service Layer (Business Logic)

The Service layer bridges the gap between the controller and data. RestaurantService.java handles:

  • Validation: Ensuring a restaurant doesn't already exist or that the owner has the correct permissions.
  • Command Processing: When searching for restaurants, it uses Java Streams or Repository methods to filter by name, location, and approval status.
  • Transaction Management: Ensures that when an order is placed, inventory is checked and order records are saved atomically.

4.4 Data Transfer Objects (DTOs)

DTOs (like RestaurantRequest, OrderResponse) are simple objects used to transfer data between the client and server. They are crucial for decoupling the internal database entities from the external API contract, allowing us to hide sensitive fields (like passwords) and aggregate data.

4.5 Model Layer (Entities)

The data models represent the database tables. Key entities include:

  • User: Stores credentials, roles (`ADMIN`, `CUSTOMER`, `OWNER`), and contact info.
  • Restaurant: Linked to a `User` (Owner) via a `@OneToOne` relationship. Contains details like address, rating, and approval status.
  • Order: The central entity linking a `User` (Customer), a `Restaurant`, and multiple `OrderItem` entities via `@OneToMany` relationships. It tracks the total price and current status.

4.6 Repository Layer

Extends JpaRepository to provide out-of-the-box CRUD operations. We also define custom query methods, such as `findByOwnerId(Long id)` or `findByCuisinesContainingIgnoreCase(String cuisine)`, allowing Spring Data JPA to automatically generate efficient SQL queries.

7. Future Integrations

While EatEasy currently provides a robust foundation for food delivery, the platform is designed with extensibility in mind. Several enhancements are planned for future iterations to improve user engagement and operational efficiency:

  • Payment Gateway Integration: Integrating third-party payment processors like Stripe or Razorpay to facilitate secure online transactions, replacing the current cash/simulated flow.
  • Real-Time Delivery Tracking: Implementing WebSocket or Firebase integration to provide live GPS tracking of delivery drivers for customers.
  • Mobile Application: Developing native mobile applications (iOS and Android) using Ionic or React Native to reach a broader audience and utilize device-specific features like push notifications.
  • AI-Driven Recommendations: Utilizing machine learning algorithms to suggest restaurants and dishes to users based on their order history and preferences.
  • Chat Support: Adding an in-app chat feature to allow direct communication between customers, restaurant owners, and support staff.

8. Conclusion

EatEasy successfully demonstrates the implementation of a modern, scalable, and secure full-stack web application. By leveraging the power of Angular for a dynamic frontend and Spring Boot for a robust backend, the project achieves a seamless user experience across multiple roles.

The system effectively handles complex relationships between users, restaurants, and orders, ensuring data integrity and secure access control. The inclusion of a comprehensive dashboard, real-time status updates, and detailed analytics provides tangible value to restaurant owners and administrators. This project serves as a solid proof-of-concept for a commercial-grade food delivery platform, with clear pathways for future scalability and feature expansion.
